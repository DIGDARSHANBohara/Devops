Deploying a Static Site to Vercel with GitHub Actions
This guide walks you through deploying a simple static site to Vercel using GitHub Actions, a powerful tool for automating deployments. This process is ideal for learning about Continuous Integration/Continuous Deployment (CI/CD) pipelines and real-world web hosting. By the end, you'll have a live site on a Vercel production domain (e.g., https://your-project.vercel.app) that updates automatically when you push changes to the main branch.
Why Vercel?
Free Tier: Sufficient for student projects, no credit card required.
Ease of Use: Automatically configures static sites with minimal setup.
Educational Value: Exposes you to industry-standard deployment pipelines.
No Server Management: Unlike GitHub Pages, Vercel is an external hosting platform, teaching you about third-party hosting.
Prerequisites
GitHub Repository: Contains your static site (HTML, CSS, JS files).
Vercel Account: Free account for deployment.
Basic Git Knowledge: You should know how to commit and push code to GitHub.
Step-by-Step Guide
1. Prepare the Static Site
Create a simple static site in your GitHub repository.
Create index.html:
Save the following file in the root of your repository or a folder like public/. For simplicity, we’ll assume it’s in the root.

 <!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Student Deployment Demo</title>
  <style>
    body { font-family: Arial, sans-serif; text-align: center; padding: 50px; }
    h1 { color: #333; }
  </style>
</head>
<body>
  <h1>Hello, GitHub Actions!</h1>
  <p>This site is deployed to Vercel automatically.</p>
</body>
</html>
Directory Structure: Place index.html in the root or public/. Ensure all site files (CSS, JS, images) are included.
Push to GitHub:
Create a new GitHub repository (e.g., static-site-demo).
Add and commit your files:
git add .
git commit -m "Add static site"
git push origin main
Set Up Vercel
Create a Vercel project to host your site.
Sign Up: Go to vercel.com and sign up using GitHub or email.
Create a New Project:
Click New Project in the Vercel dashboard.
Import your GitHub repository.
Configure the project:
Framework Preset: Select “Other” for a plain static site.
Build Settings: Leave defaults (no build command needed for static sites).
Output Directory: Set to . (root) or public/ if your site is in a subdirectory.
Deploy the project. Vercel will provide a production domain (e.g., https://your-project.vercel.app).
Get Vercel API Token:
Go to Vercel > Account Settings > Tokens.
Create a token named “GitHub Actions Deploy” and copy it.
Get Organization and Project IDs:
Organization ID: Vercel > Settings > General > ID (user or team ID).
Project ID: Vercel > Your project > Settings > General > Project ID.
Save these IDs for GitHub secrets.
Set Production Branch:
Go to Vercel > Your project > Settings > Git.
Ensure the Production Branch is set to main. This ensures pushes to main update the production domain.
3. Configure GitHub Actions
Automate deployment to Vercel’s production domain whenever you push to the main branch.
Create Workflow File:
In your GitHub repository, create a file: .github/workflows/deploy.yml
Add the following YAML configuration:
name: Deploy to Vercel Production

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install Vercel CLI
        run: npm install --global vercel@latest

      - name: Pull Vercel Environment Information
        run: |
          vercel pull --yes \
            --environment=production \
            --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}

      - name: Build Project Artifacts
        run: |
          vercel build --prod \
            --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}

      - name: Deploy Project Artifacts to Vercel
        run: |
          vercel deploy --prebuilt --prod \
            --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}

Add Secrets to GitHub:

Go to your repository > Settings > Security > Secrets and variables > Actions > New repository secret.
Add:
VERCEL_TOKEN: Your Vercel API token.
VERCEL_ORG_ID: Organization ID from Vercel.
VERCEL_PROJECT_ID: Project ID from Vercel.
4. Test the Deployment
Verify that pushing to main updates the production domain automatically.
Push Changes:

Modify index.html
Commit and push:
git add .
git commit -m "Update static site"
git push origin main
Monitor Workflow:

Go to your repository > Actions tab.
Check the “Deploy to Vercel Production” workflow status.
View logs to confirm deployment to the production domain.
Verify Production Domain:

Visit your Vercel production domain (e.g., https://your-project.vercel.app).
Confirm the changes are live without manual promotion.
In the Vercel > Deployments tab, ensure the deployment is marked as “Production” (not “Preview”).
Know the process:

GitHub Actions: Automates tasks like deployment when you push code.
Vercel: Hosts your static site and serves it on a production domain.
CI/CD: Ensures every push to main updates the live site automatically.
Production Deployment: The --prod flag and main as the production branch ensure deployments go straight to production.
Hands-On Exercise:

Modify index.html (e.g., change text or add CSS).
Push changes and monitor the GitHub Actions workflow.
Verify the updated site on the Vercel production domain.
Explore Vercel’s Deployments tab to see production vs. preview deployments.
Key Concepts:

Static Sites: Simple HTML/CSS/JS files served as-is.
Workflow YAML: Defines automation steps in GitHub Actions.
Secrets: Securely store sensitive data like API tokens.
Production vs. Preview: Pushes to main update production; other branches create preview URLs.
